## 1. 배경 — 왜 만드는가

### 1.1 문제 정의
- 네이버페이 대출비교는 **견적까지** 는 잘 한다. 하지만 견적 후 **"이 조건이 정말 내 상황에 맞는지, 실행은 어떻게 하는지"** 를 묻고 싶은 단계에서 사용자는 이탈한다.
- 은행 직원과 통화는 부담스럽고, 지점 방문은 시간이 안 맞는다. **중간 다리 역할** 이 필요하다.
- 대출모집인은 이미 존재하지만 **발견 경로가 빈약** 하다 — 은행연합회 통합조회 페이지는 행정용에 가깝고 사용자 동선이 어색하다.
- 특히 **주담대·전세 갈아타기 사용자** 는 매물 위치가 있어 "이 매물 근처에서 도와줄 사람"이라는 자연스러운 검색 의도가 발생한다.

### 1.2 가설
> "매물 주소를 가진 사용자는 그 주소 반경 N km 이내의 **해당 은행 전속 모집인** 을 만나면 실제 실행 확률이 급격히 높아진다. 네이버페이는 대출비교 결과 + 부동산 매물 + 신용점수 데이터를 모두 갖고 있어, **'이 매물 / 이 한도 / 이 모집인'** 3자를 한 화면에서 연결할 수 있는 유일한 사업자다."

### 1.3 사용자 진짜 동기

| 표면 동기 | 진짜 동기 |
|---|---|
| "대출모집인이 뭔지 보려고" | **온라인 비교만으로는 결정이 불안** — 사람에게 검증받고 싶다 |
| "근처 모집인 찾기" | **매물 계약 일정에 맞춰 만날 수 있는 사람** 이 필요 |
| "별점 높은 모집인" | **처음 만나는 사람이라 사기/푸시 영업이 두려움** |
| "○○은행 모집인만" | **대출비교 결과 1순위 은행과 같은 모집인이어야 의미 있음** |

## 2. 화면 구성 — 기능별 정리

### 2.1 글로벌 헤더 / 네비게이션
- 네이버페이 로고 + 서비스명 **"대출모집인 찾기"**
- 네비: **대출비교 · 부동산 · 모집인 찾기(활성) · 내 대출**
- **의도**: 모집인 찾기를 독립 탭으로 두되, 좌우의 **대출비교 / 부동산 / 내 대출** 탭과 데이터 연결을 전제로 함 — 견적 → 모집인, 매물 → 모집인 흐름이 양방향으로 작동.

### 2.2 지도 영역 (상단 히어로)

| 요소 | 동작 | 기획 의도 |
|---|---|---|
| 주소 검색창 | 지역·은행·모집인 이름 통합 검색 (기본값: 사용자 현재 위치) | "내가 어디서 찾을지 모를 때" 즉시 진입 |
| 지도 위 마커 | 모집인 성씨 첫 글자 (예: 박/김/이/최/정) — 클릭 시 프로필 모달 | 가독성 높고 친근, 익명성 보호 |
| 현재 매물 마커 | 본인 위치 / 검색 매물 위치 점 표시 | 거리 인지 즉각 |
| 인근 동 라벨 | 역삼동·삼성동·논현동·대치동·서초동 등 자동 표시 | 지역 맥락 파악 |
| 줌 (+/−) + 리셋(⟳) | 지도 확대/축소/원위치 | 표준 지도 조작 |
| 현재 지역 칩 | 📍 서울 강남구 역삼동 — 현재 기준 지역 표시 | 오인 방지 |

### 2.3 "대출모집인이란?" 안내 카드
- 경영진/사용자 모두에게 **제도 자체에 대한 신뢰** 를 먼저 깔아주는 영역.
- **4가지 가치 카드**: ⏰ 시간 절약 · 📋 서류 대행 · 💡 전문 상담 · 🛡️ 공인 등록제
- 출처 명시: 은행연합회 / 네이버 지식백과 — **광고가 아니라 공공 제도라는 신호**.

### 2.4 매물 주소 검색 (핵심 차별화)
- 입력: `서울 강남구 테헤란로 152` 형식 도로명 주소
- 출력: **해당 주소 기준 가까운 모집인 Top 3 + 거리(m)**
  - 예: 1. 김서윤(신한) — 85m / 2. 박준혁(KB국민) — 230m / 3. 이정민(우리) — 637m
- **기획 의도**: "내 매물 근처"라는 가장 강한 검색 의도를 잡는 자리. 부동산 탭의 매물 상세에서 **"이 매물 근처 모집인 보기"** CTA로 진입할 수 있도록 설계 예정.

### 2.5 필터 & 정렬

| 요소 | 옵션 | 의도 |
|---|---|---|
| 은행 필터 칩 | 전체 / KB국민 / 신한 / 우리 / 하나 / NH농협 / 카카오뱅크 / 토스뱅크 | 대출비교 1순위 은행과 일치시키는 자연스러운 좁히기 |
| 정렬 dropdown | 별점 높은순 (기본) / 거리순 / 후기 많은순 | 신뢰 우선 → 편의 → 검증량 순으로 옵션 제공 |
| 결과 카운트 | "내 주변 N명" | 탐색 범위 즉시 인지 |

### 2.6 모집인 리스트 카드
1장의 카드에 들어가는 정보:
- **거리** (현재 위치 또는 검색 매물 기준)
- **이름 + 은행 뱃지** (예: 박준혁 · KB국민)
- **소속 + 등록번호** (예: KB부동산신탁 강남지점 · 2024-서울-01234) — **등록번호 노출 = 공인 신호**
- **별점 + 후기 수** (예: ★★★★★ 4.9 (127))
- **전문 분야 태그** (주택담보대출 / 아파트전세 / 갈아타기 등)
- **액션 버튼**: 전화 · 문자 · 이메일 (각각 `tel:` / `sms:` / `mailto:` 또는 인앱 연결)

### 2.7 모집인 프로필 모달
리스트 카드 클릭 시 열림. 구성:

| 섹션 | 내용 |
|---|---|
| 헤더 | 이름 · 소속 · 등록번호 · "○○은행 전속 모집인" 뱃지 |
| 핵심 지표 3종 | **경력 (12년) · 누적 상담 (2,340건) · 평균 금리 (3.42%)** |
| 전문 분야 | 태그 리스트 |
| 소개 | 본인 직접 작성 — 톤·전문성 파악용 |
| 후기 | 실거래 사용자 후기 (별점 + 본문 + 날짜) 3건 이상 |
| CTA | 전화 상담 · 문자 · 이메일 |

**기획 포인트**: 평균 금리는 **객관 실적 지표** 로 신뢰 결정에 가장 강하게 작용. 단, 표본 N과 산출 기간을 함께 노출해야 함 (현재 시안에는 미반영 — 개선 필요).

## 3. 진입 시나리오 (3개 surface)

모집인 찾기는 **3개의 진입점** 에서만 제공한다. 각 진입점은 화면 컨텍스트가 다르므로 진입 시 사전 채워지는 정보가 다르다.

| # | 진입 surface | 컨텍스트 | 사전 채움 (deeplink param) |
|---|---|---|---|
| 1 | **별도 페이지** (대출모집인 찾기) | 사용자의 현재 위치 또는 직접 검색 | 없음 — 기본 진입 |
| 2 | **대출비교 결과화면** | 견적 1순위 은행 + 매물 주소(있을 시) | `?bank={code}&address={addr}` |
| 3 | **부동산 매물 상세** (대출 모듈) | 매물 주소 + 매물가 기반 LTV 한도 | `?address={addr}&propertyId={id}` |

### 3.1 진입점 ① — 별도 페이지 (자체 제작)
- **참고 시안 없음** — 본 기획서 화면 구성(§2)을 그대로 구현
- 라우트: `/loan-consultant-finder`
- 기본 진입 시 사용자 현재 위치 기반 지도 + 전체 모집인 리스트
- 진입 ②/③ 에서 들어올 경우 URL 파라미터로 **은행 필터** 또는 **매물 주소 마커** 가 사전 적용된 상태로 시작

### 3.2 진입점 ② — 대출비교 결과화면 (Figma 시안 기반)
- **참고 시안**: [Figma node 3099-19681](https://www.figma.com/design/JjBN1j67hIKGNiLs6Ki5dq/-%EC%8B%A0%EC%9A%A9%EC%A0%90%EC%88%98--%EC%95%84%EC%9D%B4%EB%94%94%EC%96%B4?node-id=3099-19681)
- **버튼 위치**: 상단 "아파트 시세" 카드 (미캘란쉐르빌 시세 15억 5,000) **바로 아래**, "신청할 수 있는 상품" 리스트 **위**
- **버튼 카피**: `이 아파트 대출 상담사에게 문의하기`
- **부카피(보조 텍스트)**: `근처 등록 모집인 N명 — 견적 1순위 ○○은행 전속`
- **클릭 시**: `/loan-consultant-finder?bank=hana&address=서울_강남구_…&from=loancompare`
  - `bank`: 견적 결과 1순위 은행 코드 자동 채움 → 도착 페이지에서 은행 필터 chip 사전 선택
  - `address`: 시세 조회한 아파트 주소 자동 채움 → 지도 중심 이동 + 가까운 모집인 Top 3 노출
- **기획 의도**: 견적까지 받은 사용자가 "이 조건이 정말 맞는지" 망설일 때 가장 정확한 타이밍. **1순위 은행 전속 모집인 = 동일 조건으로 실행 가능성 최고**.

### 3.3 진입점 ③ — 부동산 매물 상세 (Figma 시안 기반)
- **참고 시안**: [Figma node 3099-20665](https://www.figma.com/design/JjBN1j67hIKGNiLs6Ki5dq/-%EC%8B%A0%EC%9A%A9%EC%A0%90%EC%88%98--%EC%95%84%EC%9D%B4%EB%94%94%EC%96%B4?node-id=3099-20665)
- **버튼 위치**: 매물 상세 페이지 내 **"대출" 섹션** (대출 한도 / 금리 정보 두 줄이 있는 모듈). 두 줄 아래에 풀폭 버튼 추가
- **버튼 카피**: `이 매물 대출 상담사에게 문의하기`
- **부카피**: `이 매물 근처 등록 모집인 N명`
- **클릭 시**: `/loan-consultant-finder?address={매물주소}&propertyId={id}&from=realestate`
  - 도착 페이지에서 매물 위치를 지도 중심 + 마커로 자동 표시
  - 매물가 기반 LTV 계산하여 "이 매물 한도 사전 시뮬레이션" 카드를 상단에 노출 (P1)
- **기획 의도**: 매물을 진지하게 보고 있는 사용자가 가장 강한 실행 의도를 가진 단계. 대출 모듈의 "한도/금리" 정보는 견적이고, **"상담"** 이 다음 단계의 자연스러운 액션.

### 3.4 도착 페이지 공통 표준 동선
1. (deeplink 진입이면) 사전 채워진 **은행 필터 / 매물 마커** 확인
2. 지도/리스트에서 **위치·은행** 으로 추가 좁히기
3. **별점·후기·평균 금리** 로 2차 좁히기
4. **프로필 모달** 에서 경력·소개·후기 확인
5. **전화/문자/이메일** 중 부담 적은 채널로 1차 컨택

### 3.5 진입 출처 트래킹
- 모든 진입에 `from` 파라미터 필수 (`direct` / `loancompare` / `realestate`)
- KPI 측정 및 진입점별 전환율 비교 — 어느 surface 가 가장 효과적인지 판단

## 4. 비기능 요건 · 정책

### 4.1 등록·검증 정책
- 금융업협회(은행연합회) 등록 모집인만 노출. **등록번호 매일 동기화**.
- 등록 말소·정지 모집인은 노출 즉시 차단.
- 후기는 **실제 상담 이력** 이 확인된 사용자만 작성 가능. (방안: 전화/문자 클릭 → N일 후 알림으로 후기 요청)

### 4.2 노출 정책
- 같은 거리·별점일 때 **랜덤 셔플** — 특정 모집인 편중 방지.
- 유료 광고 슬롯 도입 시 반드시 **"광고" 라벨** 표기 (금감원 가이드).
- 평균 금리·누적 상담은 **표본 N + 산출 기간** 함께 노출 (현재 시안 미반영 — **P1 보완**).

### 4.3 안티-푸시 영업 보호
- "이용 Tip"의 5개 체크포인트는 **항상 노출**.
- 모집인 → 사용자 역방향 콜드콜·SMS 금지. 사용자가 먼저 컨택한 경우에만 응답 허용.
- 대출금 모집인 경유 절대 불가 — 안내 배너 상시 노출.

## 5. 성공 지표 (KPI)

| 지표 | 정의 | 1차 목표 |
|---|---|---|
| 프로필 조회율 | 리스트 노출 → 프로필 모달 오픈 | 30%+ |
| 컨택 전환율 | 프로필 오픈 → 전화/문자/이메일 클릭 | 15%+ |
| 매물 주소 검색 사용률 | 전체 진입자 중 주소 입력 비율 | 20%+ |
| 대출비교 → 모집인 유입 | 견적 결과 페이지에서 모집인 탭 클릭 | 10%+ |
| 실행 전환 (장기) | 모집인 컨택 후 N일 내 실제 대출 실행 | 측정 인프라 구축 후 설정 |

## 6. 단계별 로드맵

| 단계 | 범위 | 비고 |
|---|---|---|
| P0 (MVP) | **① 별도 페이지** (지도 + 리스트 + 프로필 + 전화/문자/메일 + 매물 주소 검색 + 은행 필터) + **② 대출비교 결과화면 진입 버튼** + **③ 부동산 매물 상세 진입 버튼** + deeplink 파라미터 처리 (`bank`, `address`, `from`) | 3 surface 동시 오픈 — 진입 채널이 충분해야 KPI 검증 가능 |
| P1 | 매물가 기반 LTV 사전 시뮬레이션 카드 (진입 ③), 평균 금리 N·기간 노출, 후기 검증 플로우, 진입 출처별 전환율 대시보드 | 차별화 지표 보강 |
| P2 | 모집인 → 사용자 견적 공유, 인앱 채팅, 신용점수 기반 한도 사전 시뮬레이션 전달 | 휴먼 ↔ 데이터 결합 |
| P3 | 광고 슬롯 / 모집인 자체 페이지 / 후기 리워드 | 수익화·생태계 |
